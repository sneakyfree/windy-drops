## ⚠️ Gaps (small; fixable in the DNA strand plan; no ADR rewrite needed)

### Gap #1 — R2 bucket name deviates from established convention
**Severity:** LOW
**Where:** ADR-053 §"Bundle storage"

ADR-053 says: `Bucket: windy-drops (per the <product>-releases R2 pattern...)`. But:
- The cited pattern from `reference_r2_desktop_distribution_pattern` is `<product>-releases` → would give `windydrops-releases`
- The dominant ecosystem convention from `reference_cloudflare_infrastructure` is `<product>-<purpose>` → e.g., `windyword-recordings`, `windycode-releases`, `windycloud-userdata` → would give `windydrops-bundles` (or `windydrops-content`)

The literal name `windy-drops` (hyphen between `windy` and `drops`) matches neither convention. Existing 10 buckets are all single-word product prefixes: `windypro-`, `windyword-`, `windyfly-`, `windychat-`, `windymail-`, `windycloud-`, `windyclone-`, `windycode-`, `eternitas-`, `windy-ecosystem-`.

**Recommendation:** Strand WD-12 (R2 bucket provisioning) specifies bucket name **`windydrops-bundles`** to match the `<product>-<purpose>` convention. Public domain stays `drops.windydrops.com` as ADR-053 already specifies. Append a one-line correction footnote to ADR-053 §"Bundle storage" when M4 lands — no full ADR delta needed; the substrate doesn't change, only the bucket name.

### Gap #2 — `/version` endpoint not called out in ADR-053 M4/M5
**Severity:** MEDIUM (MF1 is locked ecosystem policy; 12 of 13 services already comply)
**Where:** ADR-053 §"Implementation milestones" M4-M5

MF1 (Marathon Foundation #1) requires every deployed service to expose `GET /version` with the canonical shape: `{service, version, commit_sha, commit_sha_short, build_timestamp, started_at, environment}`. Reference impl: `eternitas/src/eternitas/routes/version.py` (PR #74). Currently 12 of 13 ecosystem services shipped. ADR-053 inherits ADR-013 Marathon Stack but doesn't explicitly require MF1 compliance for windy-registry.

**Recommendation:** Strand plan includes an explicit `/version` strand for windy-registry **as part of M4 repo bootstrap** (not deferred to MF1-rollout-later). Reference: `~/.claude/projects/-Users-thewindstorm/memory/reference_version_endpoint_contract.md`. No ADR delta needed — MF1 is an ecosystem-wide policy that applies automatically to every new service per ADR-013.

### Gap #3 — Webhook substrate underspecified (HMAC algorithm, header name, payload format)
**Severity:** LOW
**Where:** ADR-053 §"Webhook substrate"

ADR-053 says webhooks are "HMAC-signed with subscriber's secret" but doesn't pin:
- The HMAC algorithm (Chat uses SHA-256)
- The header name (Chat uses `X-Windy-Signature` / `X-Eternitas-Signature` family)
- The signed payload format (Chat signs raw body)
- Replay protection (Chat lacks a timestamp+nonce window currently)

Per `windy-chat/CLAUDE.md` §"Push-side Webhooks (Wave 2)", the established ecosystem pattern is:
- Header: `X-<Issuer>-Signature: sha256=<hex>` (e.g., `X-Eternitas-Signature`)
- Algorithm: HMAC-SHA256
- Payload: raw request body
- No timestamp/replay window yet — accepted by precedent

**Recommendation:** Strand WD-20 (webhook substrate) specifies:
- Header: `X-Windy-Drops-Signature: sha256=<hex>`
- Algorithm: HMAC-SHA256 over raw body
- (Future improvement: add `X-Windy-Drops-Timestamp` + 5-min replay window — out of v1 scope but reserve the header name)

No ADR delta needed; this is an implementation detail within ADR-053's existing scope.

## Additional observations (notes for the strand plan, not gaps)

### Marathon Foundations beyond MF1 to bake in from day 1
While auditing, surfaced two more MFs that should be **built into windy-registry from M4**, not deferred:

- **MF4** (per-prod-DB backup): windy-registry's Postgres needs the same backup pattern as `eternitas-prod` (s3://windy-backups-eternitas-prod with SSE-S3, versioned, 30d→90d GLACIER→expire 365d). Cheap to set up at M4; expensive to retrofit. → Strand WD-12 includes backup configuration.
- **MF7** (schema migration reversibility in CI): every Alembic migration must have a tested `downgrade()`. Caught a real bug in eternitas#75 (`sa.text("0")` boolean defaults — SQLite tolerates, Postgres rejects). → Strand WD-13 (Postgres schema) bakes downgrade tests into every migration from day 1.
